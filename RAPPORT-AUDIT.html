1. Résumé exécutif

L'application est globalement bien structurée (format de réponse API unifié, RBAC à permissions fines, refresh token httpOnly avec rotation, rate limiting, Helmet, Zod). Mais l'audit révèle des problèmes bloquants pour une mise en production sereine :

  • Le build du client ne compile plus (tsc -b échoue avec ~20 erreurs) — impossible de produire un build de production sans contourner le type-check. Le build serveur affiche lui aussi ~15 erreurs TypeScript.
  • Des fonctionnalités sont réellement cassées : l'import CSV (contacts et entreprises) et les notifications d'automatisation plantent à l'exécution sur SQLite (vérifié par test), et plusieurs modales ne s'ouvrent jamais (prop isOpen au lieu de open).
  • Le webhook VoIP est public et non signé : n'importe qui sur le réseau peut injecter ou écraser des appels dans le CRM.
  • La chaîne RBAC a deux trous structurels : le seed ne lie jamais les utilisateurs aux rôles (base fraîche = tout le monde à 403, admin compris), et le login ne renvoie pas les permissions (UI vide pour les non-admins après connexion).
  • Le déploiement est fragile : aucune migration Prisma versionnée alors que start:prod exécute prisma migrate deploy ; identifiants de démo en dur sur l'écran de login ; trust proxy absent derrière Plesk/nginx.
  • Aucun test automatisé, aucune CI, et 59 fichiers modifiés non committés au moment de l'audit.

Rien d'irrécupérable : la majorité des correctifs sont localisés et bien identifiés. Le détail action par action se trouve dans AUDIT-MODIFICATIONS.md.

2. Constats critiques

C-01 Webhook VoIP public, sans signature ni authentification Critique

POST /api/calls/webhook est volontairement monté avant authenticate et n'exige aucun secret, aucune signature HMAC, aucune restriction d'IP. De plus, le champ recording_url accepte n'importe quelle chaîne, qui est ensuite servie via res.redirect(call.recordingUrl) dans la route de streaming.

Pourquoi c'est grave : toute personne pouvant atteindre le serveur peut créer/écraser des appels (l'upsert par call_id permet d'écraser des données existantes), polluer le CRM, faire correspondre de faux appels à de vrais contacts, et faire rediriger les utilisateurs vers une URL arbitraire (phishing) lorsqu'ils écoutent un « enregistrement ».
Fichiers : server/src/routes/calls.ts:39-91 (webhook), calls.ts:328-330 (redirect)

C-02 Build client cassé : npm run build échoue Critique Vérifié

Le script de build est tsc -b && vite build et tsc -b sort en erreur (~20 erreurs). Parmi elles, de vrais bugs d'exécution : le composant Modal attend une prop open mais reçoit isOpen dans TargetsPage (2 modales) et ImportCsvModal → ces modales ne s'ouvrent jamais. Le lint relève par ailleurs 49 erreurs.

Pourquoi c'est grave : impossible de livrer une nouvelle version du front sans bidouiller, et des fonctionnalités visibles (création d'objectifs, import CSV) sont inutilisables car leurs modales ne s'affichent pas.
Fichiers : client/src/pages/targets/TargetsPage.tsx:328,342, client/src/components/ui/ImportCsvModal.tsx:88, + erreurs diverses (imports inutilisés, formatters Recharts, prop size de Spinner inexistante)

C-03 skipDuplicates plante sur SQLite : import CSV et notifications d'automatisation HS en dev Critique Vérifié à l'exécution

Prisma ne supporte pas createMany({ skipDuplicates: true }) sur SQLite : l'appel lève « Unknown argument skipDuplicates » (testé sur la base dev). Quatre endroits sont concernés : import CSV contacts, import CSV entreprises, et les deux actions NOTIFY_USER/NOTIFY_ROLE du moteur d'automatisation.

Pourquoi c'est grave : en environnement dev (SQLite), l'import CSV renvoie systématiquement une erreur 500 et aucune automatisation ne délivre jamais de notification. C'est exactement le piège SQLite/PostgreSQL documenté dans CLAUDE.md, mais non testé.
Fichiers : server/src/routes/contacts.ts:143, companies.ts:112, automation-engine.ts:142,158

C-04 RBAC structurellement cassé sur une base fraîche : seed sans roleId, pas de bypass ADMIN côté JWT Critique

Le seed crée les rôles et leurs permissions, puis crée les utilisateurs avec uniquement le champ texte role — sans jamais renseigner roleId. Or les permissions du JWT proviennent exclusivement de roleRef. Par ailleurs, le bypass ADMIN ['*'] n'existe que dans la branche « API key » du middleware : un ADMIN connecté par JWT n'a aucun traitement spécial côté serveur (alors que le client, lui, bypass role === 'ADMIN').

Pourquoi c'est grave : sur toute nouvelle installation (prod incluse), après db:seed, tous les utilisateurs — admin compris — reçoivent un JWT sans aucune permission et se font refuser (403) toutes les routes protégées. La base dev actuelle fonctionne uniquement parce que les roleId y ont été liés manuellement. L'incohérence client (bypass ADMIN) / serveur (pas de bypass) signifie aussi que l'UI peut montrer des actions que l'API refusera.
Fichiers : server/src/prisma/seed.ts:152-175, server/src/middleware/auth.ts:74-77,84-97, server/src/routes/auth.ts:61-62

C-05 Le login ne renvoie pas les permissions → UI vide pour les non-admins Critique

La réponse de POST /auth/login contient l'utilisateur sans champ permissions (elles ne sont mises que dans le JWT). Côté client, authStore.login() fait permissions: user.permissions ?? [] → tableau vide. La sidebar et tous les CanDo/usePermission filtrent sur ces permissions.

Pourquoi c'est grave : un COMMERCIAL ou TECHNICIEN qui se connecte voit une sidebar quasi vide et aucune action jusqu'à ce qu'un refresh de token survienne (l'intercepteur ne met à jour les permissions que lors d'un 401, qui n'arrive jamais puisque l'utilisateur ne peut rien appeler). Seul l'ADMIN est épargné grâce au bypass client.
Fichiers : server/src/routes/auth.ts:65-67, client/src/store/authStore.ts:34-42, client/src/lib/api.ts:42-51

C-06 Identifiants admin pré-remplis en dur sur l'écran de login Critique

LoginPage pré-remplit le formulaire avec admin@crm.local / admin123 (defaultValues) et affiche un panneau « Comptes de démo » avec trois comptes et leurs mots de passe. Le seed crée ces comptes avec des mots de passe faibles connus (admin123, password123…).

Pourquoi c'est grave : si ce build part en production (le domaine dcrm.dcb-technologies.fr est configuré), n'importe quel visiteur voit les identifiants admin pré-remplis. Et si le seed a été exécuté en prod, ces comptes existent réellement. À noter : le panneau mentionne lucas@crm.local qui n'existe même pas dans le seed (c'est jean.dupont@crm.local).
Fichiers : client/src/pages/auth/LoginPage.tsx:25,104-123, server/src/prisma/seed.ts:153-175

C-07 Aucune migration Prisma versionnée, mais start:prod exécute migrate deploy Critique

Le dossier server/src/prisma/migrations/ n'existe pas — le schéma n'a été synchronisé qu'avec db:push (sans historique). Or le script start:prod commence par prisma migrate deploy, qui n'a rien à déployer et échouera ou divergera sur une base existante.

Pourquoi c'est grave : impossible de faire évoluer le schéma en production de façon contrôlée (ni de rollback). Toute mise à jour de schéma en prod PostgreSQL devra se faire « à la main », ce qui finit toujours par une divergence dev/prod. C'est doublement risqué avec les différences SQLite/PostgreSQL déjà identifiées (C-03, E-07).
Fichiers : server/package.json:8, absence de server/src/prisma/migrations/

C-08 trust proxy absent derrière nginx/Plesk Critique

En production, Express tourne derrière le nginx de Plesk, mais app.set('trust proxy', …) n'est jamais appelé. express-rate-limit identifie alors tous les clients par l'IP du proxy.

Pourquoi c'est grave : le rate limit global (500 req/15 min) et le rate limit anti-brute-force du login (20 req/15 min) deviennent partagés entre tous les utilisateurs : 20 tentatives de login cumulées sur toute l'entreprise bloquent tout le monde pendant 15 minutes ; et inversement, le « blocage par IP » d'un attaquant ne le distingue plus des autres. Les logs (morgan) enregistrent aussi la mauvaise IP.
Fichier : server/src/index.ts:49-76

3. Constats de sévérité élevée

E-01 Modules « calls » et « apikeys » sans aucun contrôle de permission Élevé

Toutes les routes appels (liste, détail, création, modification, suppression, écoute des enregistrements) et clés API (création, révocation) ne sont protégées que par authenticate — aucune permission calls:* ou apikeys:* n'existe.

Pourquoi : n'importe quel utilisateur (TECHNICIEN inclus) peut supprimer des appels et leurs enregistrements, écouter tous les enregistrements (données sensibles RGPD), et générer une clé API permanente héritant de son rôle — y compris un ADMIN qui obtient ainsi une clé ['*'] sans expiration ni traçabilité d'usage au-delà de lastUsedAt.
Fichiers : server/src/routes/calls.ts, server/src/routes/apikeys.ts, seed des permissions (seed.ts)

E-02 Upload d'enregistrements sans validation de type de fichier Élevé

POST /calls/:id/recording accepte n'importe quel fichier jusqu'à 200 Mo, en conservant l'extension d'origine, sans vérification MIME ni magic bytes. Le fichier est ensuite servi en audio/mpeg.

Pourquoi : stockage arbitraire (exécutables, contenus illicites) sur le serveur de prod ; saturation disque (200 Mo × n appels) ; aucune permission dédiée (cf. E-01).
Fichier : server/src/routes/calls.ts:11-22,279-297

E-03 Le fallback SPA avale les routes GET /api/* inconnues Élevé

En production, app.get('*') est déclaré après les routeurs API mais avant notFound : un GET /api/nimporte-quoi renvoie index.html (HTML, 200) au lieu d'un 404 JSON.

Pourquoi : les erreurs de route côté client deviennent indétectables (le front reçoit du HTML qu'il tente de parser en JSON → erreurs trompeuses « Unexpected token < »), et le monitoring ne voit jamais de 404.
Fichier : server/src/index.ts:123-128 (utiliser un pattern excluant /api)

E-04 IDOR : marquer comme lue la notification d'un autre utilisateur Élevé

PATCH /notifications/:id/read met à jour la notification par id seul, sans filtrer sur userId (contrairement au DELETE juste en dessous, qui filtre correctement).

Pourquoi : un utilisateur peut altérer l'état des notifications d'un collègue (perte d'alertes « ticket urgent » marquées lues à son insu). Impact métier réel pour un outil de support.
Fichier : server/src/routes/notifications.ts:27-32

E-05 Remplacement des permissions d'un rôle non transactionnel Élevé

PUT /roles/:id/permissions fait deleteMany puis createMany hors transaction. Si un permissionId est invalide (contrainte FK), la création échoue après la suppression.

Pourquoi : le rôle se retrouve avec zéro permission : tous les utilisateurs du rôle perdent l'accès à l'application (leurs refresh tokens sont en plus supprimés par cette même route → déconnexion forcée vers une session sans droits).
Fichier : server/src/routes/roles.ts:161-169 (envelopper dans prisma.$transaction)

E-06 Injection de formules CSV dans tous les exports Élevé

Les exports CSV (entreprises, contacts, tickets) échappent les guillemets/virgules mais pas les cellules commençant par =, +, - ou @.

Pourquoi : un nom d'entreprise ou un titre de ticket saisi comme =HYPERLINK("http://evil";"…") ou =cmd|… s'exécute à l'ouverture dans Excel sur le poste de l'utilisateur qui exporte. Les données proviennent de saisies (et du webhook public C-01 !).
Fichiers : server/src/routes/companies.ts:136-150, tickets.ts:108-124, contacts.ts:153+

E-07 Recherche : contains sans mode: 'insensitive' → comportement différent en prod PostgreSQL Élevé

Toutes les recherches (/search, filtres tickets/contacts/companies/calls…) utilisent contains nu. Sur SQLite c'est insensible à la casse ; sur PostgreSQL c'est sensible.

Pourquoi : en production, chercher « dupont » ne trouvera plus « Dupont ». La recherche, fonction centrale d'un CRM, se dégradera silencieusement au passage en prod sans qu'aucun test ne le détecte.
Fichiers : server/src/routes/search.ts, tickets.ts, contacts.ts, companies.ts, calls.ts (toutes les clauses contains)

E-08 Scheduler : le cron horaire des automatisations se duplique à chaque rechargement Élevé

startScheduler() stoppe et recrée currentTask et reminderTask, mais le troisième cron (automatisations, '0 * * * *') est créé sans être stocké ni stoppé. Chaque sauvegarde des paramètres scheduler (PUT /settings/schedulerEnabled|schedulerTime → restartScheduler()) ajoute une exécution horaire supplémentaire.

Pourquoi : après n modifications des réglages, les automatisations tournent n+1 fois par heure : notifications en double/triple, logs d'automatisation multipliés, charge inutile.
Fichier : server/src/scheduler.ts:139-151

E-09 Vulnérabilités npm connues Élevé Vérifié

Client : 6 vulnérabilités (3 high — chaîne react-router/react-router-dom ≤ 7.14.1). Serveur : 4 modérées (express 4.21.x → qs vulnérable ; uuid < 11.1.1).

Pourquoi : correctifs disponibles via npm audit fix — dette de sécurité gratuite à résorber.
Commande : npm audit fix dans client/ et server/ puis re-test

E-10 Refresh tokens : stockage en clair, pas de détection de réutilisation, pas de purge Élevé

Les refresh tokens (valides 7 jours) sont stockés en clair dans la table RefreshToken. La rotation existe, mais une réutilisation d'un ancien token (signe de vol) n'invalide pas la famille de tokens. Aucun job ne purge les tokens expirés. Le changement de mot de passe via PATCH /users/:id/password n'invalide pas les refresh tokens existants (contrairement au reset par email qui le fait).

Pourquoi : un dump de la base donne des sessions valides 7 jours ; un token volé reste utilisable même après changement de mot de passe ; la table grossit indéfiniment.
Fichiers : server/src/routes/auth.ts, users.ts:146-170, scheduler.ts (ajouter purge)

E-11 Erreurs Prisma non mappées : 500 génériques et catch muets Élevé

Le pattern dominant est catch { res.status(500)… } sans log de l'erreur. Les erreurs Prisma prévisibles ne sont pas traduites : email dupliqué à la création d'utilisateur (P2002) → 500 au lieu de 409 ; id introuvable en update/delete (P2025) → 500 au lieu de 404 ; FK invalide (P2003) → 500 au lieu de 400.

Pourquoi : impossible de diagnostiquer un incident en prod (rien dans les logs), et le front affiche « Erreur serveur » pour des cas utilisateurs banals (email déjà pris).
Fichiers : toutes les routes ; centraliser dans middleware/errorHandler.ts + helper de mapping Prisma

E-12 Changement de mot de passe : crash 500 si currentPassword absent Élevé

Dans PATCH /users/:id/password, si l'appel omet currentPassword, bcrypt.compare(undefined, …) lève une exception → 500. Le corps n'est pas validé par Zod (seul endpoint du fichier sans schéma).

Pourquoi : erreur 500 à la place d'un message de validation propre ; incohérent avec le reste du code qui valide tout via Zod.
Fichier : server/src/routes/users.ts:146-170

E-13 Config production : CSP Helmet par défaut non testée avec le SPA, CORS LAN en prod Élevé

Helmet 8 active une Content-Security-Policy par défaut ; Express sert le build React avec ces en-têtes en prod (risque de blocage d'assets/inline selon le build, à valider). Les regex CORS autorisent toute origine 192.168.x.x / 10.x.x.x y compris en production, et les requêtes sans origin sont toujours acceptées.

Pourquoi : soit la CSP casse silencieusement des pages en prod, soit elle a été désactivée sans trace ; et la surface CORS prod devrait se limiter au domaine officiel.
Fichier : server/src/index.ts:52-65

4. Constats de sévérité moyenne

M-01 ProtectedRoute jamais utilisé : aucune garde de permission par route côté client Moyen

Le composant existe (avec support des permissions) mais App.tsx ne l'utilise pas : toutes les pages sont accessibles à tout utilisateur connecté (ex. /users, /settings/roles) ; seules les API renvoient 403, laissant des pages vides ou en erreur.

client/src/App.tsx, client/src/components/ProtectedRoute.tsx

M-02 Taxonomie des permissions incohérente Moyen

  • GET /settings (lecture) exige settings:write
  • POST /users/targets (écriture) exige reports:read
  • Les licences utilisent les permissions equipment:*
  • knowledge.ts mélange permissions et contrôles de rôle en dur (ADMIN || MANAGER)
  • GET/PUT /users/:id utilisent des checks de rôle au lieu de permissions
  • Le seed contient des permissions interventions:* qu'aucune route n'utilise
settings.ts:26, users.ts:59,76,91-134, licenses.ts, knowledge.ts:20,37, seed.ts

M-03 La recherche globale ignore les permissions par module Moyen

GET /search renvoie contacts, entreprises, tickets et opportunités à tout utilisateur authentifié, même sans pipeline:read ou contacts:read.

server/src/routes/search.ts

M-04 Moteur d'automatisation : stages « WON »/« LOST » en dur et re-notifications horaires Moyen

runOpportunityInactive exclut les stages ['WON','LOST'] en dur alors que les pipelines/stages sont dynamiques (flags isWon/isLost en base) — une opportunité gagnée dans un pipeline personnalisé sera relancée. La déduplication des tickets en retard utilise une fenêtre d'1 h pour un job qui tourne toutes les heures → re-notification potentielle à chaque exécution.

server/src/automation-engine.ts:349,296-304

M-05 Double source de vérité : User.role (texte) vs User.roleId (relation) Moyen

Le rôle existe en deux exemplaires qui peuvent diverger (cf. C-04). Les rôles personnalisés créés via l'UI ne correspondent à aucune valeur de l'enum z.enum(['ADMIN','MANAGER','COMMERCIAL','TECHNICIEN']) utilisé dans users.ts : impossible d'assigner un rôle custom à un utilisateur.

schema.prisma:21,26-27, users.ts:11-27, roles.ts

M-06 Aucun test automatisé, aucune CI, pas de lint serveur Moyen

Zéro framework de test dans les deux package.json, pas de dossier .github/, pas d'ESLint côté serveur. 59 fichiers modifiés non committés au moment de l'audit (gros risque de perte / de commit fourre-tout).

M-07 Pas de logger structuré ni de piste d'audit Moyen

Uniquement console.log/error + morgan. Aucune trace de qui a supprimé un contact, modifié un rôle, révoqué une clé API… Pour un CRM multi-utilisateur, l'absence d'audit trail rend tout incident inanalysable.

M-08 Utilisateurs désactivés irrécupérables depuis l'UI Moyen

DELETE /users/:id fait un soft delete (isActive: false) mais GET /users filtre isActive: true : un utilisateur désactivé disparaît de toutes les listes et ne peut plus être réactivé (la réactivation par PUT /users/:id existe côté API mais aucune UI ne peut l'atteindre).

server/src/routes/users.ts:30-39,137-143

M-09 RGPD : enregistrements d'appels et données personnelles sans gouvernance Moyen

Les enregistrements d'appels (données sensibles) sont conservés sans durée de rétention, accessibles à tous les utilisateurs connectés (E-01), sans journalisation des écoutes. Pas de procédure d'effacement (droit à l'oubli) pour contacts/appels. Le token de reset de mot de passe transite en query string (peut finir dans des logs).

M-10 Documentation interne non versionnée et uploads/ non ignoré Moyen

Le .gitignore exclut *.md : PLAN.md, API.md, PROGRESS.md, DESIGN.md (référencés par CLAUDE.md) n'existent que sur ce poste — perte garantie en cas de changement de machine ou de travail à plusieurs. À l'inverse, server/uploads/ n'est pas ignoré : risque de committer des enregistrements d'appels.

.gitignore

M-11 Incohérences des hooks API client Moyen

useList renvoie la réponse complète ({success, data, meta}) alors que useItem/useCreate/useUpdate renvoient data.data : chaque page doit se souvenir de la convention, source de bugs de typage (T ne décrit pas la même chose). useUpdate prend l'id à la création du hook (pattern rigide pour les listes).

client/src/hooks/useApi.ts

M-12 Webhook VoIP : correspondance de numéro fragile Moyen

Le numéro appelant est normalisé (suppression espaces/+/­tirets) puis comparé avec contains aux champs phone/mobile stockés non normalisés (ex. seed : « 04 72 12 34 56 »). « 0472123456 » ne matchera jamais « 04 72 12 34 56 » → l'auto-rattachement contact/société ne fonctionne quasiment jamais, et contains sur un numéro court peut matcher le mauvais contact.

server/src/routes/calls.ts:42-53 (stocker un champ phoneNormalized indexé)

7. Points positifs

  • Format de réponse API unifié et systématiquement respecté.
  • Refresh token httpOnly + rotation + invalidation en base (logout, reset, changement de permissions de rôle) — au-dessus de la moyenne.
  • RBAC à permissions fines avec UI de gestion des rôles ; invalidation des refresh tokens à la modification d'un rôle.
  • Validation Zod quasi systématique des corps de requêtes.
  • Rate limiting global + renforcé sur login/forgot-password ; mots de passe bcryptés (coût 12) ; réponse anti-énumération sur forgot-password.
  • Bonne hygiène secrets : .env ignoré, exemples de prod sans secrets, validation des variables d'env au démarrage.
  • Front : lazy loading des pages, chunking manuel raisonné, hooks génériques TanStack Query, aucun dangerouslySetInnerHTML, aucun console.log résiduel.
  • Transaction utilisée là où c'est le plus critique (génération de référence ticket).

8. Plan d'action recommandé

Phase 1 — Stabiliser (bloquant avant toute mise en prod)
